UI Acceptance Checklist (QA) — Preparer/Reviewer/Approver Signoff on Every Page/Tab/Sub-Tab

Use this as test cases for each module (Planning, Risk, Controls, Substantive, Completion, Reporting). Apply the same checklist to Page → Tab → Sub-Tab → Section nodes.

1) Presence & Consistency (Every Screen)

 SignoffStrip is visible on every page, tab, and sub-tab (no missing screens).

 Strip shows Preparer / Reviewer / Approver + Status badge + Open Issues + Last action.

 Strip persists on scroll (sticky) and does not overlap form controls.

 Navigation between tabs/sub-tabs keeps the correct node state (no reset/flicker).

2) Role Visibility & Clickability

As Preparer

 Preparer checkbox/button is clickable only in Draft or Returned.

 Reviewer/Approver checkboxes are visible but not clickable.

As Reviewer

 Reviewer checkbox/button is clickable only in In Review.

 Preparer checkbox is not clickable.

 Approver checkbox is not clickable.

As Approver

 Approver checkbox/button is clickable only in Reviewed.

 Preparer/Reviewer checkboxes are not clickable.

As non-team user

 SignoffStrip is view-only; no actions permitted.

3) Status Transitions (Happy Path)

 Preparer completes required fields → clicks Submit for Review → status becomes In Review, Preparer shows ✓ with name/time.

 Reviewer clicks Mark Reviewed → status becomes Reviewed, Reviewer shows ✓ with name/time.

 Approver clicks Approve → status becomes Approved, Approver shows ✓ with name/time.

 If node-locking enabled: Approver clicks Lock → status becomes Locked and lock icon appears.

4) Hard Blockers (Must Prevent Progression)

Submit for Review (Preparer)

 Missing mandatory fields blocks submit with “Why blocked” list.

 Missing required evidence blocks submit with link to the exact requirement.

 Failing tie-out (TB vs mapping/schedule) blocks submit and points to failing rows/fields.

 “Why blocked” items navigate user to the exact field/section.

Review (Reviewer)

 Reviewer cannot mark reviewed if any open reviewer notes remain unresolved.

 Reviewer cannot mark reviewed if required preparer signoff not present.

Approval (Approver)

 Approver cannot approve if reviewer signoff missing.

 Approver cannot approve if any critical engagement gates configured for that module are incomplete.

 Approver cannot approve if child nodes not reviewed/approved (parent aggregation rule).

5) Return Workflow (Round Trip)

 Reviewer can click Return to Preparer from In Review and must enter reason (required).

 Approver can click Return to Preparer from Reviewed and must enter reason (required).

 After Return, status becomes Returned and Preparer gets edit rights back.

 Returned node shows a visible Returned badge and count of return notes.

 Preparer resubmission creates a new version and preserves prior review history.

6) Segregation of Duties (SoD) Enforcement

 System blocks Reviewer = PreparedBy for the same node (hard stop).

 System blocks Approver = PreparedBy for the same node (hard stop).

 If policy disallows Reviewer = Approver: system blocks it (hard stop).

 If policy allows Reviewer = Approver: system requires exception reason, logs it, and flags “SoD Exception” in UI.

7) Edit Rights & Read-only States

Draft/Returned

 Preparer can edit substantive fields.

 Reviewer/Approver are read-only (can comment).

In Review

 Substantive edits are blocked for everyone (except comment/responses if allowed).

 Preparer cannot change content unless returned.

Reviewed

 Preparer cannot edit.

 Reviewer cannot change substantive fields (only add reviewer summary/notes if allowed).

 Approver can only Approve/Return.

Approved/Locked

 All substantive editing is blocked for everyone.

 Only configured unlock workflow exists.

8) Parent–Child Aggregation Rules

 Parent node cannot be submitted/reviewed/approved if any child is incomplete.

 Parent SignoffStrip shows drill-down list of incomplete children with direct links.

 When all children at a level are complete, parent checkbox becomes eligible.

 Status roll-ups correctly display at Page/Tab/Sub-tab level.

9) “Not Applicable (N/A)” Handling

 Preparer can mark node N/A only with reason.

 N/A still requires Reviewer + Approver signoffs to be treated as complete.

 Parent aggregation counts Approved N/A as complete.

 N/A state is clearly visible and audit-trailed.

10) Audit Trail & Traceability (Inspection-Ready)

 Every action (submit/return/review/approve/lock/unlock/N/A) creates an immutable log entry.

 Audit trail displays: who, role, timestamp, from-status → to-status, reason (where required).

 Version history shows differences (or at least version numbers + timestamps).

 AI-assisted insertions are flagged “AI-Assisted” with acceptance user/time.

11) Notifications & Tasking

 On submit, assigned Reviewer(s) receive a notification/task.

 On return, Preparer receives notification with reason and link.

 On review completion, Approver receives notification.

 Notifications include engagement, node name, due date (if configured).

12) Error Handling & Concurrency

 If two users attempt conflicting actions, system prevents overwrites (optimistic lock).

 Clear error shown if status changed by someone else (“Record updated; refresh”).

 No duplicate signoffs created on double-click / slow network.

13) Export & Reporting Gates

 PDF/export shows signoff stamps (Prepared/Reviewed/Approved with names/times).

 Report generation is blocked unless required nodes are Approved/Approved N/A.

 Any post-approval unlock triggers mandatory re-review and re-approval for impacted nodes.

14) UX Microcopy & Clarity

 Buttons are labeled consistently: Save Draft / Submit for Review / Return / Mark Reviewed / Approve / Lock.

 “Why blocked” list uses plain language and is actionable (field-level).

 Open Issues counter matches unresolved notes + missing evidence + failing tie-outs.

15) Minimum Regression Set (Run on Every Release)

 One full module end-to-end (Draft → Approved) without returns.

 One return cycle (Reviewer return + Approver return).

 One SoD violation attempt (must block).

 One parent-child gating scenario (parent blocked until all children done).

 One N/A scenario (counts only after approval).

 One unlock scenario (forces re-review and re-approval).
