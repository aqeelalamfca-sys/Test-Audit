Ultimate Implementation Prompt (UI + Workflow): Preparer / Reviewer / Approver Checkboxes on Every Page/Tab/Sub-Tab

You are the UI + workflow enforcement engine for an AI-based audit software. Implement a persistent 3-step sign-off strip on every screen level: Page → Tab → Sub-Tab → Section/Widget (where applicable). The strip must show Preparer, Reviewer, Approver as checkboxes that become green checkmarks only when the eligible user completes the required action. These sign-offs are not cosmetic: they control edit rights, status transitions, locks, audit trail, and reporting gates.

1) Universal UI Component (must render everywhere)

Create a reusable component: SignoffStrip (sticky/top-right or top ribbon), shown on:

every page

every tab

every sub-tab

every section that can be independently completed (optional but recommended)

SignoffStrip layout

Preparer ☐ / ✓

Reviewer ☐ / ✓

Approver ☐ / ✓

Current Status badge: Draft / In Review / Returned / Reviewed / Approved / Locked

Open Issues counter

Last action + timestamp

Why blocked link (shows missing items)

2) Sign-off Data Model (hierarchical)

Each UI node (page/tab/sub-tab) is a Signoff Node with a parent-child relationship.

Node Types: PAGE, TAB, SUBTAB, SECTION

Each node must store:

node_id, node_type, node_key (unique stable identifier), parent_node_id

status (same lifecycle)

prepared_by, prepared_at

reviewed_by, reviewed_at

approved_by, approved_at

locked_by, locked_at

open_issues_count

last_action, last_action_at

Aggregation rule (mandatory):

A parent node cannot show ✓ unless all child nodes meet that level’s completion.

Example: Tab “Planning” Preparer ✓ only if all sub-tabs/sections inside Planning have Preparer ✓ (or are explicitly marked “Not Applicable” with reason and approval).

3) Checkbox Behavior (role-based, status-based)
A) Preparer checkbox

Visible: always
Clickable: only if current user has Preparer role for that engagement AND node status is Draft or Returned.

When Preparer clicks “Complete/Prepared”:

Run validations:

mandatory fields filled

required attachments present (if defined)

internal tie-outs passed (if defined)

no unresolved “critical” issues at preparer level

If pass:

set node: prepared_by, prepared_at

set status: In Review

show Preparer ✓

lock substantive edits for preparer (read-only except responses/comments)

trigger notification to Reviewer(s)

If fail:

block and show “Why blocked” list on the same screen (field-level pointers)

Preparer cannot uncheck after submission. Must be returned by Reviewer/Approver.

B) Reviewer checkbox

Visible: always
Clickable: only if user has Reviewer role AND node status is In Review.

Reviewer actions

Return (preferred UI: “Return to Preparer” button next to checkbox)

requires reason + optional field references

status becomes Returned

Preparer ✓ remains recorded but node returns to editable state for preparer

Reviewer checkbox stays empty

Mark Reviewed

Confirm:

no open reviewer notes unresolved

required procedures done for that node type

preparer evidence sufficiency checks passed

Enforce SoD:

reviewer must not equal prepared_by

If pass:

set reviewed_by, reviewed_at

status → Reviewed

show Reviewer ✓

notify Approver(s)

Reviewer cannot edit preparer substantive fields; only add review notes/review summary.

C) Approver checkbox

Visible: always
Clickable: only if user has Approver role AND node status is Reviewed.

Approver actions

Return

status → Returned

requires reason + instruction

Approve

Confirm:

SoD: approver ≠ prepared_by (and ≠ reviewed_by unless policy allows with explicit exception reason)

all child nodes at same level are reviewed

no critical engagement-level blockers (if configured)

If pass:

set approved_by, approved_at

status → Approved

show Approver ✓

node becomes read-only

Lock

optional per node; if enabled:

status → Locked (strongest)

unlock requires dual-control + forces return/re-review/re-approve

Approver cannot directly change preparer content; must return.

4) “Not Applicable (N/A)” Handling (important for sub-tabs)

For any node, allow Mark N/A only with:

reason required

allowed by Preparer, but N/A becomes effective only after Reviewer + Approver sign-off

N/A nodes still appear in aggregation, but treated as “complete” once approved

5) Parent/Child Gate Logic (must be enforced)

When a user attempts to complete a parent node:

System must ensure every child is:

Prepared before parent prepared

Reviewed before parent reviewed

Approved before parent approved
OR formally Approved as N/A.

UI must show drill-down list of incomplete child nodes with direct navigation links.

6) Edit Rights (automatic and strict)

Draft/Returned: Preparer can edit; Reviewer/Approver read-only + comments

In Review: No substantive edits; only comments/responses

Reviewed: Frozen; only Approver can approve/return

Approved/Locked: Read-only; unlock is exception workflow only

7) Audit Trail (immutable, per node and per action)

Every checkbox click or status change must write an event:

node_id, from_status, to_status

action (PREPARE_SUBMIT / REVIEW_SIGNOFF / RETURN / APPROVE / LOCK / UNLOCK / NA_MARK)

performed_by, role, timestamp

reason (mandatory for return/unlock/NA)

8) Visual Rules (what the user sees)

Empty box ☐ = pending

Green check ✓ = completed

Yellow warning icon = blocked (hover shows missing items)

Red badge = returned (shows count of return notes)

Grey lock icon = locked

9) Mandatory Buttons & Microcopy (every node)

Next to SignoffStrip provide:

Save Draft (Preparer only in Draft/Returned)

Submit for Review (Preparer only)

Return to Preparer (Reviewer/Approver)

Mark Reviewed (Reviewer only)

Approve (Approver only)

View Open Issues (all roles)

10) System Messages (standardize)

When blocked, show:

“Cannot submit/review/approve because:”

bullet list of missing fields, missing evidence, unresolved notes, failing tie-outs

each item links to the exact field/section

Final Enforcement Statement (non-negotiable)

Do not allow any “overall engagement completion” or “report generation/finalization” unless:

all required nodes are Approved (or Approved N/A), and

engagement-level close checklist is Approved, and

any post-approval change triggers automatic Returned → Re-review → Re-approve for impacted nodes.
