## 5) SaaS subscription logic integration (Trial → Active → Dormant)

### Plan/Status model (firm-level)

* `trial` → `active` → `suspended` → `dormant` → `deleted`

### Enforcement logic

1. **Trial (60 days)**

   * Full access except: watermark on exports, limited file storage, limited AI credits.
2. **Trial Expired**

   * If not paid: firm becomes **dormant**
   * **Dormant** means: read-only access (view only), no edits, no new engagements, no exports.
3. **Dormant retention**

   * After retention window (e.g., 30 days) → soft-delete + scheduled hard delete per policy.
4. **Suspended**

   * Due to billing failure or compliance breach; similar to dormant but with SA messaging and admin notice.

### Subscription gating points (must be centralized middleware)

* Block write APIs for: objectives/risks/responses/policy uploads/monitoring reviews/exports when not `active` or `trial`.
* Block engagements from “issue” if not `active`.
* SA can manage billing, but **cannot query** any firm tables.

### “Firm Wide Controls” specific gating

* Exports (Quality Manual PDF, Monitoring reports) **active only** (trial gets watermark).
* Document uploads **active/trial only**.
* Independence declarations **allowed** even in dormant (read-only operations + submit allowed) if you want “reactivation readiness”.

## 6) Production security checklist (Audit-grade)

### A) Tenant isolation & authorization

* [ ] Every API endpoint enforces `firm_id` from session token; never from request body/query.
* [ ] Row-level filtering in repository/data layer mandatory (unit tested).
* [ ] SA endpoints operate only on platform tables (billing, firm status) with `firm_id = null` logs.
* [ ] Deny-by-default RBAC; explicit allow lists per endpoint.

### B) Authentication & session security

* [ ] Strong password policy (min length + uppercase + complexity as you defined).
* [ ] MFA optional for Firm Admin/Partner (recommended).
* [ ] Secure cookies (HttpOnly, Secure, SameSite=strict).
* [ ] Rate limit login + lockout after N attempts + IP-based throttling.

### C) Data protection

* [ ] Encrypt sensitive fields at rest (document keys, tokens, ethics reports).
* [ ] Object storage presigned URLs with short expiry.
* [ ] Database backups encrypted + tested restore drills.

### D) Auditability

* [ ] Immutable audit logs (append-only) with before/after JSON.
* [ ] Admin actions log actor, IP, user-agent, timestamp.
* [ ] Evidence integrity: hash stored for uploaded critical documents.

### E) AppSec

* [ ] Input validation on every endpoint (Zod/FluentValidation).
* [ ] CSRF protection on state-changing requests.
* [ ] Content Security Policy (CSP) strict.
* [ ] Dependency scanning + SAST (CI) + container image scanning.

### F) Availability

* [ ] Health checks + uptime monitor.
* [ ] Centralized structured logging + alerting.
* [ ] DB indexes for firm_id + frequent filters.
