Jira-ready QA checklist structured as executable test cases, followed by a sample users + role assignment matrix you can import into your test environment.

You can paste the test cases directly into Jira (one issue per test case), or bulk import via CSV.

PART A — Jira-Ready QA Test Cases

Module Scope: Universal SignoffStrip (Page → Tab → Sub-Tab → Section)
Applies To: All audit modules (Planning, Risk, Controls, Substantive, Completion, Reporting)

Epic: SIGNOFF GOVERNANCE ENGINE
TC-001 — SignoffStrip Presence (All Nodes)

Type: Functional
Priority: Critical
Precondition: User logged in (any role)

Steps:

Navigate to any engagement.

Open a Page.

Switch to a Tab.

Switch to a Sub-tab.

Open a nested Section.

Expected Result:

SignoffStrip visible at every level.

Displays: Preparer / Reviewer / Approver + Status badge + Open Issues + Last Action.

Sticky behavior confirmed (does not disappear on scroll).

Correct node context maintained when navigating.

TC-002 — Preparer Submit (Happy Path)

Precondition:

Logged in as Preparer

Node status = Draft

All required fields completed

Steps:

Click “Submit for Review”.

Expected Result:

Status changes to In Review

Preparer checkbox shows ✓

prepared_by and prepared_at stored

Substantive editing disabled

Reviewer receives notification

Audit trail entry created

TC-003 — Preparer Blocked (Missing Fields)

Precondition:

Logged in as Preparer

Node status = Draft

Required fields intentionally left blank

Steps:

Click “Submit for Review”.

Expected Result:

Submission blocked

“Why blocked” panel shows missing fields

Links navigate to exact missing field

No status change occurs

TC-004 — Reviewer Signoff (Happy Path)

Precondition:

Logged in as Reviewer

Node status = In Review

No open review notes

Steps:

Click “Mark Reviewed”.

Expected Result:

Status changes to Reviewed

Reviewer checkbox shows ✓

reviewed_by and reviewed_at stored

Approver notified

Audit trail entry created

TC-005 — Reviewer Return

Precondition:

Logged in as Reviewer

Node status = In Review

Steps:

Click “Return to Preparer”.

Enter reason (mandatory).

Confirm.

Expected Result:

Status changes to Returned

Return reason logged

Preparer regains edit rights

Returned badge visible

Audit trail entry created

TC-006 — Approver Approval (Happy Path)

Precondition:

Logged in as Approver

Node status = Reviewed

All children approved

Steps:

Click “Approve”.

Expected Result:

Status changes to Approved

Approver checkbox shows ✓

approved_by and approved_at stored

Node becomes read-only

Audit trail entry created

TC-007 — Approver Blocked (Child Incomplete)

Precondition:

Parent node

One child node still Draft

Steps:

Attempt to Approve parent.

Expected Result:

Approval blocked

“Why blocked” lists incomplete child node(s)

Clickable links to incomplete child

TC-008 — Segregation of Duties (SoD Block)

Precondition:

Same user assigned as Preparer and Reviewer

Steps:

Attempt to review own prepared node.

Expected Result:

System blocks action

Error message: “Segregation of Duties violation”

No status change

Audit trail logs attempted violation (optional)

TC-009 — Unlock Workflow

Precondition:

Node status = Approved

Steps:

Approver initiates Unlock.

Provide reason.

Second approver/admin confirms (if dual-control enabled).

Expected Result:

Status moves to Returned

Re-review and re-approval required

Unlock reason logged

Audit trail entry created

TC-010 — N/A Flow

Precondition:

Node not applicable

Steps:

Preparer marks node N/A with reason.

Reviewer signs off.

Approver approves.

Expected Result:

Node marked Approved N/A

Parent aggregation treats node as complete

N/A reason stored

Audit trail reflects full cycle

TC-011 — Audit Trail Integrity

Precondition: Any signoff action performed

Steps:

Open Audit Trail panel.

Expected Result:

Shows: who, role, timestamp, from-status, to-status, reason (if required)

No ability to edit/delete entries

Entries listed chronologically

TC-012 — Parent-Child Aggregation Logic

Precondition:

Sub-tab incomplete

Tab attempted for review

Steps:

Attempt to mark Tab as Prepared.

Expected Result:

Blocked

System shows list of incomplete sub-tabs

Tab remains Draft

TC-013 — Concurrency Control

Precondition:

Two users open same node

Steps:

User A submits.

User B attempts action without refresh.

Expected Result:

System shows “Record updated — refresh required”

No duplicate signoffs

Data integrity preserved
